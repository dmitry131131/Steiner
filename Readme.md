# Реализация алгоритма 1-Steiner

Данная работа посвящена реализации и оптимизации алгоритма 1-Steiner построения дерева Штейнера для заданного множества точек на плоскости. Программа принимает набор исходных координат и генерирует граф с добавленными точками Штейнера, минимизируя суммарную стоимость связей.

## Сборка и запуск

Проект использует C++20 и собирается с помощью CMake. Управление зависимостями происходит автоматически через CMake Package Manager (CPM), устанавливать их вручную не требуется.

### Требования

- `CMake`
- `Git`
- Компилятор с поддержкой `C++20` (gcc 14.1+, clang 18.0+, MSVC 17.10+ и т.д.)
- Система сборки, поддерживаемая `CMake` (ninja, make и т.д.)

### Установка

**Linux**

```bash
git clone <repository-url>
cd <project-directory>
mkdir build
cmake -G "Unix Makefiles" -S . -B build/Release -DCMAKE_BUILD_TYPE=Release
cmake --build build/Release/
```

**Windows:**

```bash
git clone <repository-url>
cd <project-directory>
mkdir build
cd ./build
cmake .. -G "Visual Studio 18 2026" -A x64
cmake --build . --config Release
```

(Пример сборки для Visual Studio 18 2026 и x86_64)

Независимо от системы, при использовании вышеописанных примеров конфигурации, итоговый исполняемый файл будет расположен в папке `build/Release`.

### Использование

В базовом варианте достаточно передать программе только входной файл. В этом случае по умолчанию применяется неоптимизированная версия алгоритма.

Пример запуска:

```bash
./steiner input.json
```

Если флаг выходного файла не указан, результат будет автоматически сохранен в той же директории, что и входной файл, под именем `<имя_входного_файла>_out.json` (в данном примере это будет файл `./input_out.json`).

## Описание алгоритма

Алгоритм 1-Steiner предназначен для поиска оптимального дерева, соединяющего все заданные точки (терминалы), с возможностью добавления новых вспомогательных узлов (точек Штейнера). Добавление таких узлов позволяет сократить общую длину всех ребер в дереве по сравнению с обычным минимальным остовным деревом (MST).

### Базовая версия

Классическая реализация алгоритма работает путем жадного перебора. Для каждой потенциальной позиции новой точки Штейнера (определяется по сетке Ханана) алгоритм пересчитывает минимальное остовное дерево и оценивает выгоду. Данный подход гарантирует корректность, но обладает высокой асимптотической сложностью, что приводит к экспоненциальному росту времени выполнения при увеличении числа узлов графа.

### Оптимизированная версия

Наиболее ресурсоёмким этапом алгоритма 1-Штейнера является перебор кандидатов (точек сетки Ханана) и оценка выигрыша от добавления каждого из них в качестве точки Штейнера. Количество кандидатов растёт квадратично относительно числа терминалов, что на больших наборах данных (сотни точек) приводит к значительному времени выполнения.

Для ускорения этой стадии была применена многопоточная обработка с использованием легковесной библиотеки `BS::thread_pool`. Вызовы функции `compute_gain()` для разных кандидатов независимы и могут выполняться параллельно. Пул потоков автоматически определяет число ядер целевой системы, что обеспечивает эффективную утилизацию вычислительных ресурсов без ручной настройки.

## Формат данных

### Входные данные

Система принимает на вход JSON файл со структурой, содержащей список вершин (`node`). Исходные вершины помечаются типом `"t"` (терминалы).

```json
{
    "node": [
        {
            "name": "p1",
            "x": 90,
            "y": 75,
            "id": 1,
            "type": "t"
        },
        {
            "name": "p2",
            "x": 49,
            "y": 22,
            "id": 2,
            "type": "t"
        }
    ],
    "edge": []
}
```

### Выходные данные

В результирующем файле структура расширяется.

1. К существующим узлам добавляется массив инцидентных ребер (`edges`).
2. Добавляются вычисленные точки Штейнера (маркируются типом `"s"`).
3. Массив `edge` заполняется описанием ребер графа (идентификатор и пара соединенных вершин).

## Анализ производительности

Тестирование производилось на примерах, расположенных в папке `examples` данного репозитория. Все тесты выполнялись на Intel Core I7 .

По результатам тестирования был проведен сравнительный анализ эффективности версий `base` и `optimized`. Обе версии корректно строят деревья одинаковой стоимости. Все измерения проводились при 10-кратном повторении алгоритма для минимизации погрешности.

Для измерения времени использовалась утилита `time`. Несмотря на то, что погрешность самого таймера крайне мала, наличие ОС, кэшей и прочих механизмов процессора вносит заметную погрешность в итоговые измерения. В задачи данной работы не входит точная оценка прироста производительности, поэтому погрешность была оценена сверху в 5%.

### Выводы

Реализованный алгоритм 1-Штейнера в манхэттенской метрике демонстрирует корректную работу на всех предоставленных тестах. Добавление параллельной обработки этапа оценки кандидатов существенно повышает производительность на больших графах, что делает алгоритм применимым для задач с сотнями терминалов без потери точности.

+ На малых наборах (до 20–30 терминалов) параллелизация не даёт заметного прироста из-за накладных расходов на синхронизацию и малого объёма работы.
+ На больших тестах (100+ точек) количество кандидатов достигает десятков тысяч, и время расчёта выигрыша сокращается практически пропорционально числу используемых ядер.

Дальнейшее ускорение возможно за счёт инкрементального обновления MST при добавлении кандидата или использования более эффективных структур данных (например, очереди с приоритетом в алгоритме Прима). Однако текущая версия уже обеспечивает хороший баланс между простотой реализации и скоростью работы.
